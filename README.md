# OpenDART Financial Analyzer (with 생성형 AI)

OpenDART API를 이용하여 기업 기준정보와 재무제표를 수집하고 SQLite 데이터베이스에 저장한 뒤, 저장된 데이터를 기반으로 재무비율, 계정별 변동 및 세무 관련 지표를 분석하는 Python 프로젝트입니다.

단순한 API 호출 코드에 그치지 않고 데이터 수집, 정규화, 저장, 조회, 분석, 콘솔 입출력의 책임을 분리하는 방향으로 설계했습니다.

현재 다음 기능을 구현했습니다.

* OpenDART 공통 API Client
* 기업 고유번호 목록 동기화
* 기업명·종목코드·기업 고유번호 검색
* 단일회사 전체 재무제표 수집
* 재무제표 응답 파싱 및 정규화
* SQLite 저장 및 중복 방지
* 저장된 재무제표 조회
* 주요 재무비율 계산
* 재무비율 계산 결과 저장 및 조회
* 계정별 당기·전기 금액 조회
* 계정별 증감액·증감률 계산
* 주요 계정 변동 이상징후 탐지
* 매출액 증가율과 재고자산 증가율 비교
* 실효세율 계산
* 이연법인세자산·부채 등 세무 관련 계정 변동 조회
* 명령별 콘솔 처리 모듈 분리
* CLI 기반 기능 실행

현재 분석 기능은 재무제표 본문에 포함된 정형 데이터를 기반으로 합니다.

세무 관련 기능 역시 실효세율과 법인세 계정의 총액 변동을 확인하는 수준이며, 구체적인 법인세 변동 원인을 분석하려면 향후 법인세 주석 데이터를 추가로 수집하고 구조화할 필요가 있습니다.

향후에는 법인세 주석 분석, 기업 간 비교, 재무비율 추세 분석, 감사보고서 수집, 핵심감사사항 분석 및 생성형 AI 기반 감사·세무 관점 분석 기능으로 확장할 계획입니다.

## 프로젝트 목적

전자공시시스템 DART는 사업보고서, 감사보고서, 재무제표 등 다양한 기업 공시 정보를 제공합니다.

하지만 공시 데이터를 실제 분석에 이용하려면 다음 작업이 필요합니다.

* 기업 고유번호 확보
* API 요청 조건 구성
* JSON·XML 응답 검증
* 계정과 금액 데이터 정규화
* 중복 데이터를 고려한 저장
* 기업·연도·보고서별 조회
* 저장된 데이터를 이용한 재무 분석
* 분석 대상 계정의 식별과 계정명 차이 처리
* 수치 변동과 원인 분석의 구분

이 프로젝트는 이러한 과정을 하나의 데이터 처리 흐름으로 구성하는 것을 목적으로 합니다.

```text
기업 기준정보 구축
        ↓
재무제표 수집
        ↓
데이터 정규화
        ↓
SQLite 저장
        ↓
재무비율 계산
        ↓
계정별 변동 및 주요 계정 분석
        ↓
실효세율 및 세무 계정 변동 조회
        ↓
향후 주석·감사보고서 연계 분석
```

재무비율과 계정 변동은 DART 응답을 받은 즉시 계산하지 않고, 데이터베이스에 저장된 재무제표를 다시 조회하여 계산합니다.

이를 통해 원본 데이터 수집과 파생 분석 결과 생성을 분리했습니다.

현재는 다음과 같은 분석 기능을 제공합니다.

* 재무비율 계산
* 재무제표 계정별 증감액·증감률 계산
* 주요 계정 간 증가율 비교
* 실효세율 계산
* 세무 관련 재무상태표 계정 변동 조회

수치상의 변동은 위험이나 오류에 대한 결론으로 사용하지 않습니다.

대신 추가 검토가 필요한 계정이나 영역을 선별하는 기초 자료로 활용하며, 실제 원인 분석에는 산업 상황, 거래 구조, 회계정책 및 재무제표 주석을 함께 검토해야 한다는 점을 전제로 합니다.

여러 계층에서 반복적으로 사용하는 금액 포맷팅, 문자열 너비 계산, 텍스트 정렬 및 입력값 변환 등의 공통 보조 기능은 프로젝트 루트의 `utils.py`에 정리했습니다.

# 주요 기능

## 1. OpenDART API Client

공통 HTTP Client가 OpenDART API 요청을 담당합니다.

주요 기능은 다음과 같습니다.

* 인증키 자동 포함
* GET 요청 처리
* JSON 응답 처리
* ZIP 파일 다운로드
* HTTP 오류 처리
* OpenDART 응답 상태 코드 검증
* 사용자 정의 예외 처리

각 서비스가 HTTP 요청 코드를 개별적으로 구현하지 않고 공통 Client를 재사용하도록 구성했습니다.

OpenDART 응답의 상태 코드를 확인하여 정상 응답과 오류 응답을 구분하며, 조회 결과가 없는 경우와 실제 요청 오류를 구분할 수 있도록 처리합니다.

예를 들어 OpenDART 상태 코드 `013`은 조회된 데이터가 없는 경우이므로, 시스템 오류로 중단하기보다 데이터가 존재하지 않는 상황으로 처리할 수 있도록 구성했습니다.

---

## 2. 기업 기준정보 동기화

OpenDART의 `corpCode.xml` 데이터를 이용하여 기업 고유번호 목록을 동기화합니다.

동기화 과정은 다음과 같습니다.

```text
ZIP 파일 다운로드
        ↓
압축 해제
        ↓
XML 파싱
        ↓
기업 정보 정규화
        ↓
SQLite 저장 또는 갱신
        ↓
기존 기업의 활성 상태 확인
```

저장 항목은 다음과 같습니다.

* 기업 고유번호
* 기업명
* 종목코드
* DART 수정일
* 활성 여부
* 최초 확인 시각
* 최근 확인 시각
* 비활성화 시각

프로그램 최초 실행 시 저장된 기업 정보가 없을 때만 전체 기업 목록을 자동으로 동기화할 수 있습니다.

이미 기업 정보가 존재하면 최초 동기화를 생략하여 불필요한 전체 동기화를 방지합니다.

기존에 저장되어 있던 기업이 최신 목록에서 확인되지 않는 경우에는 해당 기업을 삭제하지 않고 비활성 상태로 변경합니다.

이를 통해 과거 데이터와의 연결을 유지하면서 최신 기업 목록 상태도 함께 관리할 수 있습니다.

---

## 3. 기업 검색 및 선택

저장된 기업 정보를 다음 값으로 검색할 수 있습니다.

* 기업명
* 종목코드
* DART 기업 고유번호

부분 일치 검색을 지원하며 검색 결과가 여러 개인 경우 목록에서 기업을 선택합니다.

검색 결과가 하나이면 해당 기업을 자동으로 선택합니다.

검색 결과가 많으면 최대 표시 건수와 전체 검색 건수를 함께 안내합니다.

기업 선택 기능은 재무제표 수집, 재무비율 계산, 계정 변동 조회 및 세무 분석 등 여러 명령에서 공통으로 사용됩니다.

동일한 기업 검색 및 선택 로직이 각 명령에 반복되지 않도록 별도의 기업 선택 모듈로 분리했습니다.

## 4. 재무제표 수집

OpenDART의 `fnlttSinglAcntAll.json` API를 이용하여 단일회사의 전체 재무제표를 수집합니다.

지원 조건은 다음과 같습니다.

### 보고서 종류

| 입력값                             | 보고서    |
| ------------------------------- | ------ |
| `11011`, `annual`, `a`, `사업보고서` | 사업보고서  |
| `11013`, `q1`, `1q`, `1분기`      | 1분기보고서 |
| `11012`, `half`, `h`, `반기`      | 반기보고서  |
| `11014`, `q3`, `3q`, `3분기`      | 3분기보고서 |

### 재무제표 구분

| 입력값                         | 구분     |
| --------------------------- | ------ |
| `CFS`, `연결`, `연결재무제표`       | 연결재무제표 |
| `OFS`, `별도`, `개별`, `별도재무제표` | 별도재무제표 |

수집한 JSON 응답은 Parser에서 내부 저장 형식으로 정규화한 후 Repository로 전달합니다.

서비스 계층은 다음 흐름을 관리합니다.

```text
입력값 검증
        ↓
OpenDART API 요청
        ↓
응답 상태 확인
        ↓
재무제표 Parser 호출
        ↓
정규화된 데이터 반환
        ↓
Repository 저장
```

## 5. 재무제표 저장 및 조회

재무제표는 가능한 한 OpenDART 원본 구조를 유지하여 SQLite에 저장합니다.

주요 저장 항목은 다음과 같습니다.

* 접수번호
* 보고서 코드
* 사업연도
* 기업 고유번호
* 연결·별도 구분
* 재무제표명
* 재무제표 구분
* 계정 ID
* 계정명
* 계정 상세
* 당기 금액
* 전기 금액
* 전전기 금액
* 통화 단위

동일 공시의 동일 계정이 반복 저장되지 않도록 데이터베이스 제약조건과 중복 방지 로직을 적용했습니다.

재무제표 수집 결과에서는 다음 값을 확인할 수 있습니다.

* API 수신 행 수
* 실제 저장 행 수
* 중복 제외 행 수

저장된 재무제표는 기업, 사업연도, 보고서 코드, 연결·별도 구분 및 재무제표 종류에 따라 다시 조회할 수 있습니다.

조회 결과는 재무상태표, 손익계산서 및 현금흐름표 등 재무제표 구분별로 나누어 출력합니다.

## 6. 주요 재무비율 계산

저장된 재무제표를 조회하여 다음 재무비율을 계산합니다.

| 재무비율  | 계산 기준        |
| ----- | ------------ |
| 영업이익률 | 영업이익 ÷ 매출액   |
| 순이익률  | 당기순이익 ÷ 매출액  |
| ROA   | 당기순이익 ÷ 평균자산 |
| ROE   | 당기순이익 ÷ 평균자본 |
| 부채비율  | 부채총계 ÷ 자본총계  |
| 유동비율  | 유동자산 ÷ 유동부채  |

모든 결과는 백분율 단위로 계산합니다.

### 구현 특징

* 저장된 재무제표 원본 데이터를 이용
* 평균자산을 이용한 ROA 계산
* 평균자본을 이용한 ROE 계산
* 당기·전기 금액을 함께 활용
* 표준 계정 ID와 계정명 Alias 활용
* 계산이 불가능한 비율은 `None`으로 처리
* 계산 로직과 콘솔 출력 로직 분리
* 계산 결과의 분자와 분모를 함께 저장

기업마다 동일한 경제적 의미를 가진 계정의 표시명이 다를 수 있으므로, 표준 계정 ID와 계정명 후보를 함께 활용하여 필요한 계정을 찾습니다.

전기 금액이 없거나 필요한 계정을 찾을 수 없는 경우에는 임의의 값으로 대체하지 않고 계산 불가로 처리합니다.

---

## 7. 재무비율 결과 저장

계산된 재무비율은 `financial_ratio_results` 테이블에 저장합니다.

한 개의 재무비율을 한 행으로 저장하며, 주요 항목은 다음과 같습니다.

* 기업 고유번호
* 사업연도
* 보고서 코드
* 연결·별도 구분
* 비율 코드
* 비율명
* 계산 결과
* 분자 금액
* 분모 금액
* 계산 버전
* 계산 시각

계산 결과뿐 아니라 분자와 분모를 함께 저장하여 이후 계산 근거를 확인할 수 있도록 구성했습니다.

계산 버전을 별도로 저장하므로 향후 계산 공식이나 계정 매핑 기준이 변경될 경우 버전별 결과를 구분할 수 있습니다.

---

## 8. 저장된 재무비율 조회

계산과 조회 명령을 분리했습니다.

* `ratio`: 저장된 재무제표를 이용해 비율을 계산하고 저장
* `ratios`: 데이터베이스에 저장된 재무비율 결과를 조회

조회 시 다음 조건을 사용합니다.

* 기업
* 사업연도
* 보고서 코드
* 연결·별도 구분

`ratios` 명령은 재무비율을 다시 계산하거나 OpenDART API를 호출하지 않고, 현재 데이터베이스에 저장된 계산 결과만 출력합니다.

이를 통해 재무비율의 계산과 조회 책임을 분리했습니다.

---

## 9. 계정별 증감액·증감률 계산

저장된 재무제표의 당기 금액과 전기 금액을 비교하여 각 계정의 증감액과 증감률을 계산합니다.

```text
증감액 = 당기 금액 - 전기 금액

증감률 = 증감액 ÷ |전기 금액| × 100
```

전기 금액이 0이면 증감률을 계산할 수 없으므로 `None`을 반환합니다.

계산 결과에는 다음 정보가 포함됩니다.

* 계정명
* 재무제표 구분
* 당기 금액
* 전기 금액
* 증감액
* 증감률

`change` 명령을 사용하면 저장된 재무제표를 기준으로 특정 재무제표에 포함된 계정의 당기 금액, 전기 금액, 증감액 및 증감률을 확인할 수 있습니다.

증감률만 출력하지 않고 당기·전기 금액과 절대 증감액을 함께 보여주어, 비율 변화와 금액 규모를 함께 판단할 수 있도록 구성했습니다.

---

## 10. 주요 계정 변동 분석

모든 계정의 증감률을 단순 나열하는 것과 별도로, 일부 주요 계정의 변동 관계를 비교하는 분석 기능을 구현했습니다.

현재는 매출액 증가율과 재고자산 증가율을 비교합니다.

```text
재고자산 증가율 - 매출액 증가율
```

재고자산 증가율이 매출액 증가율을 일정 수준 이상 상회하는 경우 추가 검토가 필요한 변동으로 표시합니다.

현재 적용한 기준은 두 증가율의 차이가 10%포인트 이상인 경우입니다.

다만 이 결과는 재고자산의 과대계상이나 판매 부진을 직접 의미하지 않습니다.

다음과 같은 상황에서도 재고자산 증가율이 매출액 증가율을 상회할 수 있습니다.

* 향후 수요 증가에 대비한 선제적 생산
* 신규 제품 출시 준비
* 원재료 가격 상승
* 공급망 불확실성에 대비한 재고 확보
* 반도체 등 산업 사이클 변화
* 기업 인수나 사업 구조 변경

따라서 수치상의 이상징후는 결론이 아니라 추가 분석이 필요한 대상을 선별하는 기준으로 활용합니다.

---

## 11. 실효세율 계산

저장된 손익계산서 데이터를 이용하여 실효세율을 계산합니다.

```text
실효세율 = 법인세비용 ÷ 법인세차감전순이익 × 100
```

기업마다 계정명이 다를 수 있으므로 다음과 같은 계정명 차이를 고려합니다.

* 법인세비용
* 법인세비용(수익)
* 법인세차감전순이익
* 법인세비용차감전순이익

실효세율은 기업이 실제로 적용받은 세율을 단순하게 보여주는 지표이지만, 법정세율과의 차이 원인을 직접 설명하지는 않습니다.

실효세율이 높거나 낮은 원인을 분석하려면 다음 항목을 포함한 법인세 주석을 추가로 검토해야 합니다.

* 세무상 비과세수익과 손금불산입
* 세액공제
* 이월결손금
* 과거 법인세 조정
* 해외 종속기업의 세율 차이
* 이연법인세자산 미인식
* 법정세율과 실효세율의 조정 내역

현재 기능은 실효세율의 계산과 연도별 변동 확인을 위한 기초 지표를 제공하는 것을 목적으로 합니다.

---

## 12. 세무 관련 계정 변동 조회

재무상태표에 포함된 세무 관련 계정을 선별하여 당기 금액, 전기 금액, 증감액 및 증감률을 조회합니다.

현재 주요 분석 대상은 다음과 같습니다.

* 이연법인세자산
* 이연법인세부채
* 당기법인세자산
* 당기법인세부채

기업마다 계정 표시명이 다를 수 있으므로 여러 계정명 후보를 하나의 분석 계정으로 묶어 처리합니다.

예를 들어 다음 계정들은 동일한 분석 범주로 분류할 수 있습니다.

```text
이연법인세자산
비유동이연법인세자산
```

조회 결과는 증감액의 절댓값이 큰 계정을 우선적으로 확인할 수 있도록 구성했습니다.

```text
1. 이연법인세자산
   당기 금액: 18,840,559,000,000원
   전기 금액: 14,236,468,000,000원
   증감액: +4,604,091,000,000원
   증감률: +32.34%
```

다만 현재 기능은 재무제표 본문에 표시된 계정 총액의 변동만 보여줍니다.

예를 들어 이연법인세자산이 증가했다는 사실은 확인할 수 있지만, 증가 원인이 다음 중 무엇인지는 알 수 없습니다.

* 퇴직급여 관련 일시적 차이
* 재고자산 평가충당금
* 대손충당금
* 감가상각비
* 이월결손금
* 이월세액공제
* 미인식 이연법인세자산의 변동

따라서 이 기능은 법인세 변동 원인을 확정하는 분석이 아니라, 향후 법인세 주석에서 우선적으로 확인할 계정을 선별하는 스크리닝 기능으로 사용합니다.

# 콘솔 명령어

프로그램은 `main.py`에서 실행합니다.

```bash

```

실행 후 다음 명령어를 사용할 수 있습니다.

| 명령어          | 기능                         |
| ------------ | -------------------------- |

```text
차이: +6.58%p

분석 결과:
재고자산 증가율과 매출액 증가율의 차이가 설정된 기준 이내입니다.
```

재고자산 증가율이 매출액 증가율을 10%포인트 이상 상회하면 추가 검토가 필요한 변동으로 표시합니다.

다만 이 결과는 이상징후를 탐지하기 위한 스크리닝이며, 재고자산의 과대계상이나 판매 부진을 직접 의미하지 않습니다.

## 8. 실효세율 계산

```text
선택> etr

기업명, 종목코드 또는 고유번호를 입력하세요: 삼성전자
사업연도 [기본값: 2025]:
보고서 코드 [기본값: 11011]:
재무제표 구분 [기본값: CFS]:
```

출력 예시:

```text
[실효세율 계산 결과]

법인세비용: 12,345,678,000,000원
```

법인세비용이나 법인세차감전순이익 계정을 찾을 수 없거나, 법인세차감전순이익이 0인 경우에는 계산 불가로 처리합니다.

현재 기능은 실효세율을 계산하는 수준이며, 법정세율과의 차이 원인은 법인세 주석을 추가로 확인해야 합니다.

# 주요 설계 결정

## 1. API 응답과 저장 로직 분리

OpenDART API 응답을 Repository에 직접 전달하지 않고 Parser를 거쳐 내부 저장 형식으로 변환합니다.

```text
OpenDART API 응답
        ↓
Parser
        ↓
내부 저장 형식
        ↓
Repository
        ↓
SQLite
```

이 구조를 통해 다음 효과를 기대할 수 있습니다.

* API 응답 형식 변경에 대한 대응
* 저장 구조 변경의 영향 범위 축소
* Parser와 Repository의 독립 테스트
* 외부 데이터와 내부 저장 구조 간 결합도 감소
* 숫자형 데이터와 누락값의 일관된 변환

---

## 2. 원본 데이터와 파생 데이터 분리

재무제표는 API에서 수집한 원본 데이터에 가깝게 저장하고, 재무비율 및 계정 변동은 별도의 분석 서비스에서 계산합니다.

```text
financial_statements
        │
        ├── financial_ratio_service
        ├── account_change_ratio_service
        ├── major_account_analysis
        └── tax_analysis_service
```

이를 통해 재무제표 수집 로직을 변경하지 않고 새로운 분석 기능을 추가할 수 있습니다.

또한 분석 로직에서 문제가 발견되더라도 원본 재무제표를 다시 수집하지 않고 저장된 데이터를 이용해 계산 로직만 수정하고 재실행할 수 있습니다.

---

## 3. 계산과 조회 분리

재무비율 계산 명령과 저장 결과 조회 명령을 분리했습니다.

```text
ratio
    계산 → 저장 → 출력

ratios
    저장 결과 조회 → 출력
```

조회 과정에서 자동으로 재계산하지 않기 때문에 사용자는 현재 저장된 결과와 새로 계산한 결과를 명확히 구분할 수 있습니다.

향후 재무제표 정정공시를 반영할 때는 재무제표 갱신과 파생 분석 결과 재계산을 별도의 절차로 관리할 예정입니다.

---

## 4. 계산 근거 저장

재무비율 값만 저장하지 않고 계산에 사용한 분자와 분모를 함께 저장합니다.

이를 통해 다음 사항을 확인할 수 있습니다.

* 어떤 금액을 이용해 비율을 계산했는지
* 계정 매핑이 적절했는지
* 계산 버전 변경 전후 결과가 어떻게 달라졌는지
* 재무제표 정정 이후 어떤 값이 바뀌었는지

분석 결과를 단순히 출력하는 데서 끝내지 않고, 결과가 만들어진 근거를 다시 확인할 수 있도록 구성했습니다.

---

## 5. 재실행 가능성과 중복 방지

동일한 기업과 공시 데이터를 반복 수집하더라도 중복 데이터가 계속 쌓이지 않도록 설계했습니다.

재무비율 결과 역시 기업, 연도, 보고서, 재무제표 구분, 비율 코드 및 계산 버전을 기준으로 저장 또는 갱신할 수 있도록 구성했습니다.

이 구조를 통해 동일한 작업을 반복 실행하더라도 데이터베이스 상태가 불필요하게 증가하지 않도록 했습니다.

---

## 6. 명시적인 계산 불가 처리

필요한 계정이나 전기 데이터가 존재하지 않는 경우 임의의 값이나 0으로 대체하지 않습니다.

계산할 수 없는 결과는 `None`으로 유지하고 콘솔에서는 `계산 불가`로 표시합니다.

이는 다음 두 상황을 구분하기 위한 처리입니다.

* 실제 금액이 0인 경우
* 데이터가 없거나 계산 조건을 충족하지 못한 경우

데이터 부족을 실제 0으로 처리하면 비율과 증감률이 왜곡될 수 있으므로 두 상태를 명확히 구분합니다.

---

## 7. 계정 ID와 계정명 후보의 병행 사용

OpenDART 재무제표에는 표준 계정 ID가 포함되지만, 모든 기업이 동일한 계정 ID와 계정명을 사용하는 것은 아닙니다.

따라서 분석에 필요한 계정을 찾을 때 다음 기준을 함께 사용합니다.

* 표준 계정 ID
* 대표 계정명
* 계정명 Alias
* 기업별 표시명 차이

예를 들어 법인세차감전순이익은 기업에 따라 다음과 같이 표시될 수 있습니다.

```text
법인세비용차감전순이익
법인세차감전순이익
법인세비용차감전계속사업이익
```

계정명 후보를 관리하여 표시 방식의 차이로 인해 계산이 누락되는 문제를 줄이고자 했습니다.

---

## 8. 수치 탐지와 원인 분석의 분리

계정 변동이나 실효세율의 변화만으로 위험 또는 오류를 확정하지 않습니다.

현재 분석 기능은 다음 역할을 수행합니다.

```text
수치 변동 탐지
        ↓
추가 검토 대상 선별
        ↓
주석·산업 상황·거래 구조 확인
        ↓
원인 판단
```

예를 들어 이연법인세자산이 크게 증가했다는 결과는 확인할 수 있지만, 그 원인이 세액공제인지 이월결손금인지 일시적 차이인지는 재무제표 본문만으로 판단하기 어렵습니다.

따라서 현재 구현된 세무 분석은 결론을 제시하기보다 법인세 주석에서 우선 확인할 영역을 선별하는 데 목적이 있습니다.

---

## 9. 콘솔 명령 모듈 분리

기능이 추가되면서 `controller.py`에 입력 처리, 기업 선택, 분석 실행 및 결과 출력 코드가 집중되는 문제가 발생했습니다.

이를 개선하기 위해 명령별 처리 로직을 별도의 모듈로 분리했습니다.

```text
controller.py
        │
        ├── corporation_commands
        ├── financial_statement_commands
        ├── financial_ratio_commands
        └── tax_commands
```

`controller.py`는 명령어를 해석하고 적절한 명령 객체를 호출하는 역할을 중심으로 유지합니다.

기업 검색과 선택처럼 여러 명령에서 공통으로 사용하는 기능도 별도의 선택 모듈로 분리하여 중복을 줄였습니다.

---

## 10. 공통 보조 기능 분리

금액 포맷팅, 한글 문자열 너비 계산, 출력 정렬 및 입력값 변환과 같이 여러 계층에서 반복되는 보조 기능은 프로젝트 루트의 `utils.py`에 정리했습니다.

이를 통해 분석 또는 콘솔 모듈이 출력 형식과 관련된 세부 구현을 반복하지 않도록 했습니다.

# 설치 및 실행

## 1. 저장소 복제

```bash
git clone https://github.com/sangbongc/opendart-financial-analyzer.git
cd opendart-financial-analyzer
```

## 2. 가상환경 생성 및 실행

Windows:

```bash
python -m venv .venv
.venv\Scripts\activate
```

macOS 또는 Linux:

```bash
python -m venv .venv
source .venv/bin/activate
```

## 3. 패키지 설치

```bash
pip install -r requirements.txt
```

## 4. 환경변수 설정

`.env.example`을 복사하여 `.env` 파일을 생성합니다.

Windows:

```bash
copy .env.example .env
```

macOS 또는 Linux:

```bash
cp .env.example .env
```

`.env`에 OpenDART API 인증키를 입력합니다.

```env
DART_API_KEY=your_api_key
```

실제 인증키는 Git 저장소에 포함하지 않습니다.

## 5. 프로그램 실행

```bash
python main.py
```

# 향후 개발 방향

## 1. 법인세 주석 분석

현재 세무 관련 기능은 재무제표 본문에 표시된 총액을 기준으로 다음 정보를 제공합니다.

* 실효세율
* 이연법인세자산 변동
* 이연법인세부채 변동
* 당기법인세자산·부채 변동

하지만 총액 변동만으로는 구체적인 원인을 설명할 수 없습니다.

예를 들어 이연법인세자산이 증가한 경우에도 그 원인은 다음과 같이 다양할 수 있습니다.

* 퇴직급여 관련 일시적 차이
* 재고자산 평가충당금
* 대손충당금
* 감가상각비 차이
* 미사용 세무상 결손금
* 미사용 세액공제
* 이연법인세자산 인식 여부 변경

향후에는 사업보고서의 법인세 관련 주석을 수집하고, 다음 표를 구조화하는 기능을 구현할 계획입니다.

* 법인세비용의 구성
* 법정세율과 실효세율의 조정 내역
* 이연법인세자산·부채의 발생 원천
* 이월결손금
* 이월세액공제
* 미인식 이연법인세자산
* 기간별 변동 내역

주석에서 추출한 구성 항목의 합계와 재무제표 본문의 총액을 대사하여 데이터의 완전성과 일관성을 확인하는 기능도 검토하고 있습니다.

---

## 2. 주요 계정 변동 분석 고도화

현재는 매출액 증가율과 재고자산 증가율을 비교합니다.

향후에는 다음 계정 관계를 추가로 분석할 수 있습니다.

* 매출액과 매출채권
* 매출액과 재고자산
* 매출액과 매입채무
* 영업이익과 영업현금흐름
* 유형자산과 감가상각비
* 매출채권과 대손충당금
* 법인세차감전순이익과 법인세비용

단순히 증가율이 크다는 이유만으로 위험으로 판단하지 않고, 계정 간 관계와 금액 규모를 함께 고려하는 분석 구조를 목표로 합니다.

수치상의 이상징후는 결론이 아니라 추가 분석이 필요한 대상을 선별하는 기준으로 활용합니다.

---

## 3. 기업 간 비교

같은 사업연도와 보고서 조건을 기준으로 여러 기업의 재무비율을 비교하는 기능을 추가할 예정입니다.

비교 시 다음 요소를 함께 고려할 수 있습니다.

* 동일 업종 여부
* 연결·별도 재무제표 구분
* 사업연도와 보고서 종류
* 기업 규모
* 계정 표시 방식 차이
* 일회성 손익의 영향

향후 업종별 또는 사용자가 선택한 기업군을 기준으로 재무비율과 주요 계정 변동을 비교할 수 있도록 확장할 계획입니다.

---

## 4. 추세 분석

현재 분석은 하나의 사업연도에서 당기와 전기를 비교하는 방식입니다.

향후에는 여러 사업연도의 데이터를 연결하여 다음과 같은 추세를 확인할 수 있도록 확장할 계획입니다.

* 재무비율 장기 추세
* 주요 계정 증감률 추세
* 실효세율 추세
* 이연법인세자산·부채 추세
* 반복적으로 발생하는 이상징후

단년도 수치만으로 판단하기 어려운 변동을 여러 기간의 흐름에서 검토하는 기능을 목표로 합니다.

---

## 5. 감사보고서 분석

감사의견, 강조사항, 계속기업 관련 불확실성 및 핵심감사사항 등 비정형 텍스트를 수집하고 분석하는 기능을 검토하고 있습니다.

향후에는 다음 흐름으로 연결할 계획입니다.

```text
재무제표 수치 분석
        ↓
주요 변동 계정 선별
        ↓
감사보고서 및 핵심감사사항 수집
        ↓
관련 계정과 감사 절차 연결
        ↓
추가 검토사항 제시
```

예를 들어 재고자산 변동이 크게 나타난 기업의 핵심감사사항에서 재고자산 평가가 주요 감사 영역으로 언급되었는지 확인할 수 있습니다.

---

## 6. 생성형 AI 활용

향후 생성형 AI는 재무 수치를 직접 확정하거나 감사·세무 결론을 자동으로 내리는 도구보다 다음 작업을 보조하는 방향으로 활용할 계획입니다.

* 법인세 주석 요약
* 핵심감사사항 요약
* 관련 재무제표 계정 연결
* 주요 변동의 추가 검토사항 제안
* 여러 기간의 변동 원인 후보 정리
* 분석 결과의 자연어 보고서 작성

정형 데이터의 계산 결과와 생성형 AI의 해석 결과를 구분하여 제시하는 구조를 목표로 합니다.

예를 들어 계산 결과와 AI 보조 해석을 다음과 같이 나눌 수 있습니다.

```text
[정형 데이터 계산 결과]
이연법인세자산 증가율: +32.34%

[추가 검토가 필요한 항목]
법인세 주석의 이월세액공제, 이월결손금 및 일시적 차이 구성 확인

[생성형 AI 보조 해석]
주석 내용을 바탕으로 변동 원인 후보와 관련 회계·세무 검토사항 요약
```

최종 판단에는 원본 공시 확인과 전문가 검토가 필요하다는 전제를 유지합니다.

# 주의사항

* 본 프로젝트는 학습 및 포트폴리오 목적으로 개발했습니다.
* OpenDART API 이용 정책과 호출 제한을 준수해야 합니다.
* 분석 결과는 투자 권유, 세무 자문 또는 회계·감사 의견을 의미하지 않습니다.
* 재무비율은 저장된 재무제표를 기반으로 계산되므로 원본 데이터의 수집 시점과 정정 여부를 함께 확인해야 합니다.
* 정정공시 이후에는 관련 재무제표와 파생 분석 결과를 다시 갱신해야 할 수 있습니다.
* 기업마다 계정 ID와 계정명이 다를 수 있으므로 계정 매핑 결과를 추가로 검토해야 합니다.
* 계정 변동률이 크다고 해서 곧바로 회계 오류나 위험을 의미하지는 않습니다.
* 실효세율은 법인세비용과 법인세차감전순이익을 이용한 단순 계산 결과이며, 법정세율과의 차이 원인은 법인세 주석을 확인해야 합니다.
* 이연법인세자산·부채 변동 결과는 총액 변동을 보여주는 스크리닝 결과이며, 구체적인 원인 분석을 위해서는 주석 상세 내역이 필요합니다.
* 생성형 AI 기반 분석을 추가하더라도 원본 공시와 계산 결과를 우선적으로 확인해야 합니다.
* 실제 OpenDART API 인증키는 저장소에 포함하지 않습니다.
