# OpenDART Financial Analyzer (with 생성형 AI)

OpenDART API를 이용하여 기업 기준정보와 재무제표를 수집하고 SQLite 데이터베이스에 저장한 뒤, 저장된 데이터를 기반으로 재무비율과 계정별 변동을 분석하는 Python 프로젝트입니다.

단순한 API 호출 코드에 그치지 않고 데이터 수집, 정규화, 저장, 조회, 분석, 콘솔 입출력의 책임을 분리하는 방향으로 설계했습니다.

현재 다음 기능을 구현했습니다.

* OpenDART 공통 API Client
* 기업 고유번호 목록 동기화
* 기업명·종목코드·기업 고유번호 검색
* 단일회사 전체 재무제표 수집
* 재무제표 응답 파싱 및 정규화
* SQLite 저장 및 중복 방지
* 주요 재무비율 계산
* 재무비율 계산 결과 저장 및 조회
* 계정별 증감액·증감률 계산 로직
* CLI 기반 기능 실행

향후에는 기업 간 비교, 주요 계정 변동 분석, 감사보고서 수집, 핵심감사사항 분석 및 생성형 AI 기반 감사 관점 분석 기능으로 확장할 계획입니다.

---

## 프로젝트 목적

전자공시시스템 DART는 사업보고서, 감사보고서, 재무제표 등 다양한 기업 공시 정보를 제공합니다.

하지만 공시 데이터를 실제 분석에 이용하려면 다음 작업이 필요합니다.

* 기업 고유번호 확보
* API 요청 조건 구성
* JSON·XML 응답 검증
* 계정과 금액 데이터 정규화
* 중복 데이터를 고려한 저장
* 기업·연도·보고서별 조회
* 저장된 데이터를 이용한 분석

이 프로젝트는 이러한 과정을 하나의 데이터 처리 흐름으로 구성하는 것을 목적으로 합니다.

```text
기업 기준정보 구축
        ↓
재무제표 수집
        ↓
데이터 정규화
        ↓
SQLite 저장
        ↓
재무비율 및 계정 변동 분석
        ↓
향후 감사보고서 연계 분석
```

재무비율은 DART 응답을 받은 즉시 계산하지 않고, 데이터베이스에 저장된 재무제표를 다시 조회하여 계산합니다. 이를 통해 원본 데이터 수집과 파생 분석 결과 생성을 분리했습니다.

# 주요 기능

## 1. OpenDART API Client

공통 HTTP Client가 OpenDART API 요청을 담당합니다.

주요 기능은 다음과 같습니다.

* 인증키 자동 포함
* GET 요청 처리
* JSON 응답 처리
* ZIP 파일 다운로드
* HTTP 오류 처리
* OpenDART 응답 상태 코드 검증
* 사용자 정의 예외 처리

각 서비스가 HTTP 요청 코드를 개별적으로 구현하지 않고 공통 Client를 재사용하도록 구성했습니다.

---

## 2. 기업 기준정보 동기화

OpenDART의 `corpCode.xml` 데이터를 이용하여 기업 고유번호 목록을 동기화합니다.

동기화 과정은 다음과 같습니다.

```text
ZIP 파일 다운로드
        ↓
압축 해제
        ↓
XML 파싱
        ↓
기업 정보 정규화
        ↓
SQLite 저장 또는 갱신
        ↓
기존 기업의 활성 상태 확인
```

저장 항목은 다음과 같습니다.

* 기업 고유번호
* 기업명
* 종목코드
* DART 수정일
* 활성 여부
* 최초 확인 시각
* 최근 확인 시각
* 비활성화 시각

프로그램 최초 실행 시 저장된 기업 정보가 없을 때만 전체 기업 목록을 자동으로 동기화할 수 있습니다. 이미 기업 정보가 존재하면 최초 동기화를 생략할 수 있도록 구성했습니다.

---

## 3. 기업 검색

저장된 기업 정보를 다음 값으로 검색할 수 있습니다.

* 기업명
* 종목코드
* DART 기업 고유번호

부분 일치 검색을 지원하며 검색 결과가 여러 개인 경우 목록에서 기업을 선택합니다.

검색 결과가 하나이면 해당 기업을 자동으로 선택합니다. 검색 결과가 많으면 최대 표시 건수와 전체 검색 건수를 함께 안내합니다.

---

## 4. 재무제표 수집

OpenDART의 `fnlttSinglAcntAll.json` API를 이용하여 단일회사의 전체 재무제표를 수집합니다.

지원 조건은 다음과 같습니다.

### 보고서 종류

| 입력값                             | 보고서    |
| ------------------------------- | ------ |
| `11011`, `annual`, `a`, `사업보고서` | 사업보고서  |
| `11013`, `q1`, `1q`, `1분기`      | 1분기보고서 |
| `11012`, `half`, `h`, `반기`      | 반기보고서  |
| `11014`, `q3`, `3q`, `3분기`      | 3분기보고서 |

### 재무제표 구분

| 입력값                         | 구분     |
| --------------------------- | ------ |
| `CFS`, `연결`, `연결재무제표`       | 연결재무제표 |
| `OFS`, `별도`, `개별`, `별도재무제표` | 별도재무제표 |

수집한 JSON 응답은 Parser에서 내부 저장 형식으로 정규화한 후 Repository로 전달합니다.

---

## 5. 재무제표 저장

재무제표는 가능한 한 OpenDART 원본 구조를 유지하여 SQLite에 저장합니다.

주요 저장 항목은 다음과 같습니다.

* 접수번호
* 보고서 코드
* 사업연도
* 기업 고유번호
* 연결·별도 구분
* 재무제표명
* 재무제표 구분
* 계정 ID
* 계정명
* 계정 상세
* 당기 금액
* 전기 금액
* 전전기 금액
* 통화 단위

동일 공시의 동일 계정이 반복 저장되지 않도록 데이터베이스 제약조건과 중복 방지 로직을 적용했습니다.

재무제표 수집 결과에서는 다음 값을 확인할 수 있습니다.

* API 수신 행 수
* 실제 저장 행 수
* 중복 제외 행 수

---

## 6. 주요 재무비율 계산

저장된 재무제표를 조회하여 다음 재무비율을 계산합니다.

| 재무비율  | 계산 기준        |
| ----- | ------------ |
| 영업이익률 | 영업이익 ÷ 매출액   |
| 순이익률  | 당기순이익 ÷ 매출액  |
| ROA   | 당기순이익 ÷ 평균자산 |
| ROE   | 당기순이익 ÷ 평균자본 |
| 부채비율  | 부채총계 ÷ 자본총계  |
| 유동비율  | 유동자산 ÷ 유동부채  |

모든 결과는 백분율 단위로 계산합니다.

### 구현 특징

* 저장된 재무제표 원본 데이터를 이용
* 평균자산을 이용한 ROA 계산
* 평균자본을 이용한 ROE 계산
* 당기·전기 금액을 함께 활용
* 표준 계정 ID와 계정명 Alias 활용
* 계산이 불가능한 비율은 `None`으로 처리
* 계산 로직과 콘솔 출력 로직 분리
* 계산 결과의 분자와 분모를 함께 저장

전기 금액이 없거나 필요한 계정을 찾을 수 없는 경우에는 임의의 값으로 대체하지 않고 계산 불가로 처리합니다.

---

## 7. 재무비율 결과 저장

계산된 재무비율은 `financial_ratio_results` 테이블에 저장합니다.

한 개의 재무비율을 한 행으로 저장하며, 주요 항목은 다음과 같습니다.

* 기업 고유번호
* 사업연도
* 보고서 코드
* 연결·별도 구분
* 비율 코드
* 비율명
* 계산 결과
* 분자 금액
* 분모 금액
* 계산 버전
* 계산 시각

계산 결과뿐 아니라 분자와 분모를 함께 저장하여 이후 계산 근거를 확인할 수 있도록 구성했습니다.

계산 버전을 별도로 저장하므로 향후 계산 공식이나 계정 매핑 기준이 변경될 경우 버전별 결과를 구분할 수 있습니다.

---

## 8. 저장된 재무비율 조회

계산과 조회 명령을 분리했습니다.

* `ratio`: 저장된 재무제표를 이용해 비율을 계산하고 저장
* `ratios`: 데이터베이스에 저장된 재무비율 결과를 조회

조회 시 다음 조건을 사용합니다.

* 기업
* 사업연도
* 보고서 코드
* 연결·별도 구분

`ratios` 명령은 재무비율을 다시 계산하거나 OpenDART API를 호출하지 않고, 현재 데이터베이스에 저장된 계산 결과만 출력합니다.

이를 통해 재무비율의 계산과 조회 책임을 분리했습니다.

---

## 9. 계정별 증감률 계산

저장된 재무제표의 당기 금액과 전기 금액을 비교하여 각 계정의 증감액과 증감률을 계산하는 서비스 로직을 구현했습니다.

```text
증감액 = 당기 금액 - 전기 금액

증감률 = 증감액 ÷ |전기 금액| × 100
```

전기 금액이 0이면 증감률을 계산할 수 없으므로 `None`을 반환합니다.

계산 결과에는 다음 정보가 포함됩니다.

* 계정명
* 재무제표 구분
* 당기 금액
* 전기 금액
* 증감액
* 증감률

현재 서비스 계층의 계산 로직을 구현한 상태이며, 주요 계정 선별 및 CLI 출력 기능은 후속 개발 대상으로 두고 있습니다.

# 콘솔 명령어

프로그램은 `main.py`에서 실행합니다.

```bash
python main.py
```

실행 후 다음 명령어를 사용할 수 있습니다.

| 명령어      | 기능                     |
| -------- | ---------------------- |
| `help`   | 명령어 목록 출력              |
| `sync`   | DART 기업 고유번호 목록 동기화    |
| `find`   | 기업명·종목코드·고유번호로 기업 검색   |
| `fs`     | 기업 재무제표 수집 및 저장        |
| `ratio`  | 저장된 재무제표로 재무비율 계산 및 저장 |
| `ratios` | 저장된 재무비율 조회            |
| `exit`   | 프로그램 종료                |

## 기본 사용 흐름

```text
sync
  ↓
find
  ↓
fs
  ↓
ratio
  ↓
ratios
```

```text
선택> find
```

```text

기업명, 종목코드 또는 고유번호를 입력하세요: 삼성전자

보고서 코드 [기본값: 11011]:
재무제표 구분 [기본값: CFS]:
```

예상 출력 형식:

```text

```

```text

재무제표 구분 [기본값: CFS]:
```

`ratios` 명령은 지정한 조건과 일치하는 저장 결과만 조회합니다.

# 데이터베이스

현재 주요 테이블은 다음과 같습니다.

## `dart_corporations`

기업 기준정보를 저장합니다.

| 컬럼               | 설명            |
| ---------------- | ------------- |
| `corp_code`      | DART 기업 고유번호  |
| `corp_name`      | 기업명           |
| `stock_code`     | 상장회사 종목코드     |
| `modify_date`    | DART 정보 수정일   |
| `is_active`      | 현재 기업목록 활성 여부 |
| `first_seen_at`  | 최초 확인 시각      |
| `last_seen_at`   | 최근 확인 시각      |
| `deactivated_at` | 비활성화 확인 시각    |

## `financial_statements`

# 주요 설계 결정

## 1. API 응답과 저장 로직 분리

OpenDART API 응답을 Repository에 직접 전달하지 않고 Parser를 거쳐 내부 저장 형식으로 변환합니다.

이 구조를 통해 다음 효과를 기대할 수 있습니다.

* API 응답 형식 변경에 대한 대응
* 저장 구조 변경의 영향 범위 축소
* Parser와 Repository의 독립 테스트
* 외부 데이터와 내부 모델 간 결합도 감소

---

## 2. 원본 데이터와 파생 데이터 분리

재무제표는 API에서 수집한 원본 데이터에 가깝게 저장하고, 재무비율은 별도의 분석 서비스에서 계산합니다.

```text
financial_statements
        │
        ▼
financial_ratio_service
        │
        ▼
financial_ratio_results
```

이를 통해 재무제표 수집 로직을 변경하지 않고 분석 기능을 추가할 수 있습니다.

---

## 3. 계산과 조회 분리

재무비율 계산 명령과 저장 결과 조회 명령을 분리했습니다.

```text
ratio
    계산 → 저장 → 출력

ratios
    저장 결과 조회 → 출력
```

조회 과정에서 자동으로 재계산하지 않기 때문에 사용자는 현재 저장된 결과와 새로 계산한 결과를 명확히 구분할 수 있습니다.

향후 재무제표 정정공시를 반영할 때는 재무제표 갱신과 파생 분석 결과 재계산을 별도의 절차로 관리할 예정입니다.

---

## 4. 계산 근거 저장

재무비율 값만 저장하지 않고 계산에 사용한 분자와 분모를 함께 저장합니다.

이를 통해 다음 사항을 확인할 수 있습니다.

* 어떤 금액을 이용해 비율을 계산했는지
* 계정 매핑이 적절했는지
* 계산 버전 변경 전후 결과가 어떻게 달라졌는지

---

## 5. 재실행 가능성

동일한 기업과 공시 데이터를 반복 수집하더라도 중복 데이터가 계속 쌓이지 않도록 설계했습니다.

재무비율 결과 역시 기업, 연도, 보고서, 재무제표 구분, 비율 코드 및 계산 버전을 기준으로 저장 또는 갱신할 수 있도록 구성했습니다.

---

## 6. 명시적인 계산 불가 처리

필요한 계정이나 전기 데이터가 존재하지 않는 경우 임의의 값이나 0으로 대체하지 않습니다.

계산할 수 없는 결과는 `None`으로 유지하고 콘솔에서는 `계산 불가`로 표시합니다.

이는 데이터 부족과 실제 0 값을 구분하기 위한 처리입니다.

# 향후 개발 방향

## 주요 계정 변동 분석

매출액, 영업이익, 순이익, 재고자산, 매출채권 등 주요 계정의 증감률을 비교할 예정입니다.

단순히 증감률이 크다는 이유만으로 위험으로 판단하지 않고, 산업 상황과 기업의 경제적 실질을 추가로 검토해야 하는 분석 구조를 목표로 합니다.

예를 들어 재고자산 증가율이 매출 증가율을 크게 상회하면 다음 항목을 추가 검토할 수 있습니다.

* 판매 둔화 가능성
* 재고 진부화 가능성
* 재고자산 평가 적정성
* 신규 제품 또는 생산 확대 영향
* 산업 사이클 변화

수치상의 이상징후는 결론이 아니라 추가 분석이 필요한 대상을 선별하는 기준으로 활용할 계획입니다.

## 기업 간 비교

같은 사업연도와 보고서 조건을 기준으로 여러 기업의 재무비율을 비교하는 기능을 추가할 예정입니다.

## 감사보고서 분석

감사의견, 강조사항, 계속기업 관련 불확실성, 핵심감사사항 등 비정형 텍스트를 수집하고 분석하는 기능을 검토하고 있습니다.

## 생성형 AI 활용

향후 생성형 AI는 재무 수치를 직접 확정하는 도구보다 다음 작업을 보조하는 방향으로 활용할 계획입니다.

* 핵심감사사항 요약
* 관련 재무제표 계정 연결
* 주요 변동의 추가 검토사항 제안
* 분석 결과의 자연어 보고서 작성

정형 데이터의 계산 결과와 생성형 AI의 해석 결과를 구분하여 제시하는 구조를 목표로 합니다.
